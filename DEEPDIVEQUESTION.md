📚 Cyber Finance: 백엔드 설계 및 아키텍처 심화 과제

본 과제는 현재 개발 중인 주식 관리 시스템의 기술적 한계를 분석하고, 실무적인 해결 방안을 모색하기 위한 서술형 문항입니다.

1. 동기적 지연과 스레드 관리

현재 StockController는 주식 하나당 약 1.1초의 대기 시간을 가집니다. 만약 관리 대상 주식이 50개로 늘어나고 동시 접속자가 100명이 되었을 때, 서버의 Thread Pool과 사용자 응답 속도에 어떤 영향을 미칠지 서술하세요.

내 생각 / 조사결과
1.1초 * 50개 * 100명 = 약 5500초 = 약 1.5시간의 대기 시간이 발생
심각한 수준의 서버 대기 상태가 발생한다.
그러나 그 코드에 대기 시간을 넣은것은 Alpha Vantage API의 Rate Limit을 피하기 위한 조치이고 테스트 코드이므로 실제 운영 환경에서는 필요하지 않을것이다.

Gemini의 해설
정확한 계산입니다. 톰캣(Tomcat) 같은 서블릿 컨테이너는 보통 200개 내외의 스레드 풀을 가집니다. 한 요청이 스레드를 55초간 점유하면, 단 몇 명의 접속만으로 전체 스레드가 고갈되어 서버가 '먹통'이 됩니다. 실무에서는 **비동기 호출(WebClient)**이나 **캐싱(Redis)**을 통해 사용자가 대기하지 않게 설계합니다.

2. 보안: Password Hashing & Salt

사용자 비밀번호를 평문으로 저장하는 것은 보안상 치명적입니다. 단순히 SHA-256 해시를 사용하는 것보다 BCrypt와 같은 전용 알고리즘이 Rainbow Table Attack을 방어하는 데 왜 더 유리한지 논하세요.

내 생각 / 조사결과
SHA-256는 암호를 256bit로 해싱하는 것으로 비가역적 성질을 가진다.
하지만 암호->해시값이 고정함수로서 패턴을 알아낼 수 없지만 해시 값을 미리 계산한 테이블 값을 가지고 역추적을 통한 해킹이 가능하다.
BCrypt는 salting과 키 스트레칭을 이용한다.
salting이란 기존 해싱 함수에 추가적인 랜덤한 값을 추가하여 해시값을 변경하는 기법이다.
키 스트레칭은 해싱을 반복하여 해시값을 변경하는 기법으로 컴퓨터 성능 향상에 대처한다.
이는 Rainbow Table Attack을 방지한다.

Gemini의 해설
완벽합니다. SHA-256은 설계 목적 자체가 '빠른 데이터 무결성 확인'이기에 공격자가 초당 수억 번의 대조를 할 수 있습니다. 반면 BCrypt는 의도적으로 연산 비용을 높여(Work Factor) 하드웨어 성능 향상에 대응하고, 같은 비밀번호라도 유저마다 다른 Salt를 부여해 보안성을 극대화합니다.

3. 데이터 무결성: Soft Delete vs Hard Delete

StockMaster 삭제 시 연관된 UserStock 데이터로 인해 발생하는 무결성 에러를 해결하기 위해, 데이터를 실제로 지우지 않는 Soft Delete 방식이 금융 시스템에서 왜 필수적인지 감사(Audit) 관점에서 설명하세요.

내 생각 / 조사결과
금융 시스템에서의 데이터는 매우 중요하며 유저가 관련 금융 정보의 소유권을 지녔다면 그 데이터는 바로 삭제되어서는 안된다. 따라서 Hard Delete 방식보다는 Soft Delete 방식을 통해 데이터를 삭제하더라도 복구를 할 수 있는 가능성을 열어두어야 한다.

Gemini의 해설
맞습니다. 이를 **'감사 추적(Audit Trail)'**이라고 합니다. "누가, 언제, 왜 삭제했는가"는 금융권 규제(Compliance) 사항입니다. 또한 데이터가 물리적으로 사라지면 해당 데이터와 연결된 과거 통계나 리포트가 모두 깨지는 연쇄 장애가 발생하므로 Soft Delete는 기본 사양에 가깝습니다.

4. 계층 간 책임 분리 (Separation of Concerns)

환율 가공이나 외부 API 응답 처리 로직이 특정 계층에 집중되지 않을 경우 발생하는 문제점을 지적하고, Service 레이어가 비즈니스 로직의 중심이 되어야 하는 이유를 서술하세요.

내 생각 / 조사결과
외부 API 응답 처리 로직은 데이터를 불러오는 기능으로서 분산될 시 코드의 복잡도가 증가할 수 있다. Service 레이어라는 것은 유저의 요청에 따라 정해지는 계층으로 유저의 필요에 따라 비즈니스 로직이 결정된다.

Gemini의 해설
적절합니다. 특히 외부 API 응답 데이터 포맷이 변경될 때, Service 레이어에서 이를 가공해 주면 Controller나 Entity는 수정할 필요가 없습니다. 즉, **"변화의 영향 범위를 최소화"**하는 것이 계층 분리의 핵심 목적입니다.

5. 트랜잭션의 원자성 (Atomicity)

'주식 매수' 기능 구현 시 잔고 감소와 보유 수량 증가가 하나의 논리적 단위로 묶여야 합니다. 어느 한쪽만 성공했을 때 발생할 수 있는 데이터 오류와 이를 방지하기 위한 @Transactional의 역할을 기술하세요.

내 생각 / 조사결과
주식 매수와 같은 기능은 하나의 논리로 엮여야만 하며 아닐 시 큰 문제를 야기한다.
예를 들어 서버에서는 유저에게 돈을 빼갔다고 인식하지만 유저에게는 돈이 빠져나가지 않는 현상이 발생할 수도 있다. 그렇다면 이를 악용해서 서버에 돈을 빼갔다고 인식만 시키고 다시 돈을 돌려받으면 유저에 계좌에는 돈이 쌓이기만 하는 오류가 생길수도 있다.
이를 막기 위한 @Transactional의 역할은 데이터베이스 작업을 하나의 논리로 묶어, 예외 처리가 발생할 경우 롤백을 통해 데이터베이스 작업을 취소하는 역할을 한다.

Gemini의 해설
훌륭한 비유입니다. 이를 'All or Nothing' 원칙이라고 합니다. 스프링은 Proxy 패턴을 통해 메서드 시작 시 트랜잭션을 열고, 예외 발생 시 DB 상태를 메서드 호출 전으로 되돌립니다. 금융 시스템에서 이 원자성이 깨지면 시스템 전체의 신뢰도가 붕괴됩니다.

7. 장애 전파 방지 (Circuit Breaker)

외부 API(Alpha Vantage)가 다운되거나 응답이 비정상적으로 늦어질 때, 우리 서버 전체가 마비되는 것을 막기 위한 Fallback(대체값) 전략의 중요성을 서술하세요.

내 생각 / 조사결과
서버는 유지보수가 이뤄지더라도 계속 구동하도록 하는것이 제일 중요하기 때문에 외부 API의 다운에 대비한 Fallback 전략이 필요하다. 대채값은 주로 오류 문구, 이전 값들 재사용, 기본값 사용 등이 있다.

Gemini의 해설
실무적인 답변입니다. 이를 **'Graceful Degradation(우아한 성능 저하)'**라고 부릅니다. 핵심 기능(로그인 등)은 유지하되, 부가 기능(실시간 시세)만 잠시 제한함으로써 시스템 전체의 붕괴(Cascading Failure)를 막는 전략입니다.

내 생각 / 조사결과


Gemini의 힌트

9. 스케줄링과 배치(Batch) 처리

매번 실시간 API를 호출하는 대신, 특정 주기(예: 1분)마다 전 종목 시세를 업데이트하여 DB에 저장해두는 방식이 서버 부하와 Rate Limit 해결에 어떤 도움을 주는지 논하세요.

내 생각 / 조사결과
매번 실시간으로 API 호출 시 서버 부하가 증가한다. 따라서 특정 주기마다 API를 호출하여 DB에 저장해두고 일정 주기에 따라 주식 정보를 한번에 업데이트 하는 방식이 더 적절하다.

Gemini의 힌트
Rate Limit: 유저 1,000명이 동시에 접속해서 각각 API를 호출하는 것과, 서버가 1분에 한 번만 호출해서 DB에 저장해둔 것을 1,000명이 나눠 보는 것 중 어느 쪽이 안전할까요?

데이터 일관성: 모든 유저가 '동일한 시각'의 가격 데이터를 보고 정렬하는 것이 서비스 운영 면에서 유리하지 않을까요?

10. 환경 설정의 외부화 (Externalized Configuration)

API Key와 같은 민감 정보를 소스 코드에 하드코딩하지 않고 환경 변수나 별도 설정 파일로 관리해야 하는 이유를 보안과 CI/CD(배포 자동화) 관점에서 서술하세요.

내 생각 / 조사결과
별도 설정 파일로 분리 시 이점이 많다. 우선 소스 코드에 주요 정보가 들어있지 않아 보안에도 안전하며 배포 시 설정 파일 변경 시에도 소스 코드를 변경할 필요 없이 설정 파일만 간단히 수정하여 배포 가능하다.

Gemini의 힌트
보안: .gitignore에 넣었더라도 실수로 apiKey가 포함된 설정 파일을 Public Repo에 push 했다면 어떤 일이 벌어질까요?

CI/CD: 개발용 DB 주소와 실제 운영용 DB 주소가 다를 때, 코드 수정 없이 실행 환경(OS 환경 변수 등)만 바꿔서 서버를 띄울 수 있는 방법은 무엇일까요?
